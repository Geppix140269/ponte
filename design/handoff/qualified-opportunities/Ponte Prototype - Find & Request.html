Prototype notes

One thin working journey — Find → request — built mobile-first at 390 × 844 on the frozen Brand Book v5 system, reusing the Qualified Opportunities board and detail components. Illustrative data only; no company shown has been checked and nothing is a live feed.

Interaction map

  1. Intent entry — "What do you want to do?" · four actions. Find an opportunity is wired end-to-end.
  2. Find — product + Buy/Sell/Service + optional destination; advanced filters under Refine.
  3. Transformation — the query resolves into structured search chips (400–700 ms).
  4. Results — compact context bar + first result in view; Qualified Opportunities and Market Signals in separate lanes.
  5. Detail — blueprint mobile hierarchy: kind/intent → title + last-confirmed → three prominent facts → primary action → full facts → 3 trust indicators → expandable evidence → secondary actions + persistent bottom action bar.
  6. Request — role, supply, fit; a request-readiness bar fills as answers complete.
  7. Account gate — appears only on Send; states the exact reason; entered answers preserved.
  8. Resume — request submits automatically after the gate; named "Introduction request sent" outcome.
  9. Workspace — "What needs your attention?" with the new request as the priority next action.

State preserved: your query, filters, result and scroll position survive opening a detail and returning; your request answers survive the account gate.

Motion inventory

MotionWhereMeaning
ResolveTransformation chipsAmbiguous query becomes a structured search
RevealResult cards, prominent factsInformation becomes available, by content class
AdvanceReadiness barRequest completeness progresses
Settle"Sent" acknowledgementA completed action, restrained gold/green — not the bridge

The full bridge crossing is reserved for an accepted introduction and is deliberately absent from request submission. All motion runs once, and collapses under prefers-reduced-motion.

Assumptions & open product decisions

  • The hero result is a Sell offer because the visitor is Buying — a buyer seeks sellers. Counter-intent matching logic to be confirmed.
  • Only Find an opportunity is wired; the other three intents are placeholders for later prototype routes (P1, P3, P4).
  • Minimum account fields (email, name, business, country) assumed per blueprint §11.3 — real identity providers TBD.
  • Result relevance, sort and advanced-filter logic are illustrative; ranking rules are a product decision.
  • "4 new relevant results" and counts are illustrative, not a live feed.
  • Desktop here centres the mobile canvas; the full side-by-side desktop adaptation (blueprint §17) is a later stage.
